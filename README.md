# Лабораторная работа 3

## Создание инфраструктуры через OpenStack Heat и Jenkins

### Цель работы

Автоматизировать создание инфраструктуры с использованием 
**OpenStack Heat** и интегрировать процесс развёртывания 
инфраструктуры в Jenkins.

### Задачи

В рамках лабораторной работы необходимо:

* создать репозиторий для хранения кода управления инфраструктурой;
* реализовать **Heat-шаблон (YAML)** для создания сервера;
* перенести создание инфраструктуры в Jenkins;
* развернуть инфраструктуру, запустив задачу Jenkins;
* вручную установить необходимое программное обеспечение на сервере.

## Выполнение работы

Для автоматического создания инфраструктуры был разработан 
**Heat-шаблон**, описывающий виртуальную машину в OpenStack.

Шаблон параметризован и содержит следующие основные параметры:

* **image** — образ операционной системы (`ubuntu-22.04`);
* **flavor** — тип виртуальной машины (`m1.small`);
* **network** — сеть OpenStack (`students-net`);
* **key_name** — SSH-ключ для доступа к серверу;
* **volume_size** — размер диска сервера (10 GB).

В шаблоне используется механизм `block_device_mapping_v2`, 
который создаёт **boot volume** из образа. Таким образом сервер 
загружается не напрямую с образа, а с созданного тома (volume).  
Размер тома задаётся параметром `volume_size`.  
При удалении сервера том также удаляется (`delete_on_termination: true`).

Шаблон возвращает **IP-адрес созданного сервера** через `outputs`

---
Для автоматизации создания инфраструктуры был реализован 
**Jenkins pipeline**, который использует **OpenStack CLI**.

На сервере Jenkins Node был установлен пакет:
`python3-heatclient`
что позволяет управлять стеком OpenStack через CLI.

Основные шаги pipeline:

1. Создание или обновление стека 
2. Ожидание завершения создания инфраструктуры
3. Получение IP-адреса сервера
4. Проверка доступности сервера по SSH



После создания виртуальной машины на сервере вручную 
были установлены необходимые инструменты:

* Docker
* Docker Compose

Пользователь `ubuntu` был добавлен в группу `docker`, 
чтобы запускать контейнеры без использования `sudo`.

---
Таким образом реализован подход **Infrastructure as Code**, 
позволяющий автоматически и воспроизводимо создавать 
инфраструктуру для развёртывания приложения.
